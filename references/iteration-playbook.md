# イテレーションプレイブック

15回のイテレーションで段階的に品質を向上させる。
各イテレーションは特定のフォーカスエリアに集中し、チャーンを防ぐ。

## 各イテレーションの実行手順

```
1. READ    progress/iteration-log.md を読み、前回までの学びを確認
2. FOCUS   上表で今回のフォーカスエリアを確認
3. OBSERVE preview_screenshot（desktop 1280x800 + mobile 375x812）
           preview_inspect で関連CSSプロパティを測定
4. JUDGE   evaluation-rubric.md の該当次元でスコア付け
5. PRESCRIBE 具体的な改善点を3〜5個リストアップ
6. ACT     Edit ツールで対象ファイルを修正（最小限の変更）
7. BUILD   npm run build（失敗したら修正、3回まで）
8. VERIFY  preview_screenshot で改善を確認
9. COMMIT  git commit -m "improve: iteration {N} - {フォーカス}"
10. LOG    progress/iteration-log.md に以下を追記:
           - イテレーション番号
           - フォーカスエリア
           - 変更内容（箇条書き）
           - 各次元のスコア
           - 「DO NOT REVERT」マーカー（成功した変更に付与）
```

## アンチチャーンルール

以下のルールに従い、イテレーション間の無駄な往復を防ぐ。

### ルール1: 前回保護
前回のイテレーションで変更したファイルは、追加的変更のみ許可。
既存の変更を打ち消す修正は禁止。

### ルール2: ログ必読
iteration-log.md を読まずにイテレーションを開始してはならない。
「DO NOT REVERT」マーカーが付いた変更は絶対に元に戻さない。

### ルール3: 影響確認
コンポーネント（Header, Footer, SectionHeading, CtaSection, Layout）を
変更する場合、そのコンポーネントを使用している全ページを確認する。

### ルール4: 集中
1回のイテレーションで変更するファイルは最大5ファイルまで。
フォーカスエリア外の変更は次のイテレーションに回す。

### ルール5: データとプレゼンの分離
コンテンツ（src/data/*.ts）とレイアウト（src/pages/*.astro）は
同じイテレーションで同時に大幅変更しない。

---

## 早期終了条件

以下のいずれかを満たした場合、15回を待たずに終了してよい。

1. **全次元パス**: 全5次元が閾値以上のスコアを達成
2. **改善停滞**: 2回連続でどの次元もスコアが上がらない
3. **ビルド失敗**: 3回連続でビルドが通らない（ロールバックして終了）
